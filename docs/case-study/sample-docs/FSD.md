# Functional Specification Document: Chargeback Tracker MVP

## Introduction

This Functional Specification Document (FSD) outlines the systems, screens, user interfaces, data structures, and business rules for the Chargeback Tracker MVP.

This document translates the approved Product Requirements Document (PRD) into clear, actionable technical specifications. The target audience includes front-end and back-end engineers, QA testers, and product designers.

### Scope Boundaries
* **In-Scope:** Automated read-only sync of active Stripe disputes; manual dispute entry as a fallback path; deadline tracking engine; local interactive reason-code checklists (with generic-fallback); deep-linking to Stripe; 48-hour email fallback notification (Could-have); basic win-rate reporting from 90-day post-submission polling.
* **Out-of-Scope:** Direct file/evidence upload to our servers; write-back actions to Stripe API; support for non-Stripe payment processors; historical dispute analytics beyond 90 days; multi-currency aggregation.

---

## System Overview

The Chargeback Tracker MVP is a lightweight B2B web application. It connects to Stripe via OAuth to retrieve open dispute details, accepts manually-entered disputes as a first-class alternative, manages dispute-specific workflow states within its own local database, and provides tools to guide Operations Associates through the manual resolution process in their native Stripe Dashboard.

*(Stack: Next.js + TypeScript deployed on Vercel; Postgres for persistence; Vercel Cron for the daily sync and email jobs. No Redis; no separate Ruby service.)*

### Dispute State Lifecycle Engine [FSD-REQ-002]
Disputes exist in two state-spaces: their **Stripe API State** and their **Local App State**. The App maps Stripe state changes to our interface, allowing users to override status fields with operational workflow states.

| Stripe API `status` | Local Custom Status | State Transition Trigger | UI Behavior / Rule |
| :--- | :--- | :--- | :--- |
| `needs_response` | `Open` | Dispute discovered during sync **or created via manual entry**. | Default state. Display checklist and countdown. |
| `needs_response` | `Drafting` | Manual toggle by user. | Saves progress. Updates visual badge to Orange. |
| `needs_response` | `Ready to Submit` | User toggles state manually or checks 100% of checklist items. | Changes badge to Green. Displays "Ready" indicator. |
| `under_review` | `Submitted` | Sync detects Stripe state transitioned away from `needs_response`. | Locks checklist. Countdown replaced with "Under Review" timestamp. |
| `won` | `Won` | Sync detects dispute was won. | Moves dispute to completed/history tab. Feeds win-rate reporting. |
| `lost` | `Lost` | Sync detects dispute was lost. | Moves dispute to completed/history tab. Feeds win-rate reporting. |

*Manually-entered disputes follow the same lifecycle — their `status_stripe` is always `needs_response` on creation and can only transition if the user later connects Stripe OAuth and the sync engine matches by `stripe_dispute_id`. If Stripe is never connected, the manual dispute stays in the local lifecycle only.*

## Functional Requirements

### 1. Stripe OAuth Connection [FSD-REQ-006]
* **OAuth Scope Request:** Trigger Stripe Connect flow with a strict `r_disputes` read-only access level.
* **Token Management:** Securely store access tokens and linked account metadata (Stripe Merchant Account ID, Business Display Name, Timezone) in our local database. Tokens are stored encrypted at rest (encryption method deferred to implementation — use whatever the deployment platform's KMS-equivalent provides).
* **Redirection Lifecycle:** On callback completion, initiate immediate asynchronous background import before displaying the empty state dashboard.

### 2. Dispute Synchronization Engine [FSD-REQ-007]
* **Execution Frequency:** Run background sync automatically for all active connected accounts daily at 02:00 UTC, via a Vercel Cron trigger.
* **On-Demand Sync:** Limit manual "Sync Now" clicks to once every 15 minutes per merchant. **Enforcement:** compare `now()` against `stripe_connections.last_manual_sync_at`; reject with HTTP 429 if within the window. Update `last_manual_sync_at` on every successful sync. **No Redis** — a single Postgres column is sufficient.
* **Data Processing Logic:**
    * Call the Stripe API `/v1/disputes` endpoint.
    * Loop through items where `status == needs_response`.
    * If local record exists: Update `status`, `amount`, and `due_by_timestamp`. Preserve local checklist progress and status flags if Stripe still states `needs_response`.
    * If local record does not exist: Write record, create matching default checklist items for that specific dispute's `reason` code.
    * If local record exists as `needs_response` but sync reveals Stripe's status has updated to `under_review`, `won`, or `lost`: Update local status, clear countdown, and archive the record.

### 3. Countdown Timer Engine [FSD-REQ-008]
* **Timer Resolution:** Dynamic countdown client calculations happen every 60 seconds relative to the browser's system time.
* **Server-authoritative time:** Client display uses `server_now - client_now` offset (computed on page load) to render the countdown relative to server time, so a mis-set client clock cannot mislead the operator.
* **Target Reference:** Target UTC time is parsed from the Stripe payload path: `evidence_details.due_by` (Unix epoch timestamp format), or from the user-entered timestamp for manual disputes.
* **Styling Logic Matrix:**

| Remaining Time | Display Format | Color | Badge Indicator |
| :--- | :--- | :--- | :--- |
| Less than 24 hours | `Xh Ym` (e.g., `14h 22m`) | Red, bold | `[URGENT]` |
| 24 hours to 5 days | `Xd Yh` (e.g., `2d 04h`) | Orange | None |
| 5 days or more | `X days` (e.g., `8 days`) | Gray | None |
| 0 seconds or less | `Expired` | Light Gray | `[EXPIRED]` |

### 4. Interactive Checklist Engine [FSD-REQ-009]
* **Dynamic Generation:** Instantiate local records for checklist points mapped to the dispute's `reason` payload value.
* **Storage:** Store the boolean state (`true`/`false`) of each specific checklist item mapped to `user_id` and `stripe_dispute_id`.
* **Standard Templates:** Ensure templates exactly match the PRD criteria:

| Reason Code Key | Display Title | Checklist Content |
| :--- | :--- | :--- |
| `product_not_received` | Product Not Received | Retrieve carrier tracking number & status; export proof of delivery matching checkout address; copy fulfillment notification email chain |
| `unrecognized` | Unrecognized Transaction | Locate IP address & match to billing country geo-location; export order confirmation timestamp email record; search order history for proof of past successful payments |
| `subscription_canceled` | Canceled Subscription | Export subscription TOS checkbox confirmation snapshot; find cancellation timestamp log from Stripe/database; export platform access logs proving post-cancellation inactive usage |
| `merchandise_not_as_described` | Not as Described | Capture store product description screenshot; find checkout-approved refund policy terms; gather proof customer did not initiate a valid return or refused delivery |
| *Wildcard fallback* | **Generic Evidence Checklist** | Export order receipt / invoice PDF; gather direct customer communication emails; attach delivery / fulfillment proof; attach refund-policy screenshot as shown at checkout |

*Reason codes not present in this table default to the Generic Evidence Checklist. The detail slide-out shows a dropdown allowing the user to switch to any specific template if they know their code maps to a listed category.*

### 5. Stripe Deep-Linking Utility [FSD-REQ-010]
* **Primary Trigger:** A click on the CTA button: **"Go to Stripe Dashboard to Submit"**.
* **Link Target Construction (standard accounts):** `https://dashboard.stripe.com/disputes/{{stripe_dispute_id}}`.
* **Connect Account Handling:** If user metadata returns a managed sub-account ID (Connect platform configuration), construct the deep link with the merchant context: `https://dashboard.stripe.com/{{stripe_merchant_account_id}}/disputes/{{stripe_dispute_id}}`.
* **Security Parameter:** Force deep-links to launch in a new browser tab using `target="_blank"` and `rel="noopener noreferrer"`.

### 6. Email Notification Engine [FSD-REQ-011]
* **Schedule:** Runs once daily at 08:00 in the merchant's stored timezone (`stripe_connections.timezone`, default `UTC`), via a Vercel Cron trigger.
* **Selection query:** Fetch active disputes where the time remaining until `due_by_timestamp` is within one hour of the 48-hour mark. The two-hour window catches any dispute crossing the 48h mark on any given day, given the job runs once daily.
* **Idempotency:** Track `last_48h_email_sent_at` on each dispute row; skip if an email has already been sent for this dispute in the previous 24 hours. Prevents duplicate emails from any accidental cron double-fires.
* **Email Body Content:**
    * **Subject:** `Action Required: Chargeback Deadlines in 48 Hours`
    * **Body:** Table listing Stripe ID, dispute reason, disputed amount, and a direct application link (deep-link into our own app, not into Stripe).

### 7. Win-Rate Reporting [FSD-REQ-022] *(NEW)*
* **Data source:** Disputes where `status_stripe IN ('won', 'lost')`, updated by the ongoing sync engine within the 90-day polling window post-submission (see PRD Open Question 2).
* **Computation (server-side):** Aggregate wins vs. losses grouped by `reason` code. Sum `amount_cents` for wins → total recovered. Sum for losses → total lost. Win rate = wins / (wins + losses) × 100, rounded to nearest whole percent.
* **Rendering:** Single-page dashboard with three top-line stats and a horizontal bar chart. Bars are rendered as inline SVG — no chart library dependency.
* **Empty state:** If there are no won/lost disputes yet, show a friendly explainer: *"Once you've submitted a few disputes, come back to see your win rate."*

### Table Definitions

#### Table: `accounts`
Stores top-level subscriber credentials.
* `id` (UUID, Primary Key)
* `email` (VARCHAR, Unique, Indexed)
* `created_at` (TIMESTAMP)

#### Table: `stripe_connections`
Encapsulates secure API parameters for each merchant profile.
* `id` (UUID, Primary Key)
* `account_id` (UUID, FK → `accounts.id`)
* `stripe_user_id` (VARCHAR, Unique) — Stripe merchant ID (e.g., `acct_1032EP2e`). NULLABLE — a merchant may exist with manual disputes only, no Stripe connection.
* `access_token` (VARCHAR, Encrypted) — OAuth bearer token.
* `business_name` (VARCHAR)
* `timezone` (VARCHAR, default `'UTC'`) — merchant's local timezone for email scheduling.
* `last_manual_sync_at` (TIMESTAMP, NULLABLE) — used to rate-limit the "Sync Now" button per FSD-REQ-007.
* `active` (BOOLEAN, default `true`) — flipped to `false` if Stripe returns 401/revoked auth per FSD-REQ-017.

#### Table: `disputes`
Houses high-level dispute metadata. All personal identifiers (names, cards) are omitted, keeping the database zero-PII compliant.
* `id` (UUID, Primary Key)
* `stripe_connection_id` (UUID, FK → `stripe_connections.id`)
* `stripe_dispute_id` (VARCHAR, Unique, Indexed)
* `amount_cents` (INTEGER)
* `currency` (VARCHAR)
* `reason` (VARCHAR) — Stripe reason code; may be unmapped, in which case the generic checklist is used.
* `status_stripe` (VARCHAR) — `needs_response`, `under_review`, `won`, `lost`.
* `status_local` (VARCHAR) — `Open`, `Drafting`, `Ready to Submit`, `Submitted`.
* `due_by_timestamp` (TIMESTAMP)
* `card_brand` (VARCHAR)
* `card_last4` (VARCHAR, length 4)
* `source` (VARCHAR) — `'stripe'` for OAuth-synced or `'manual'` for user-entered.
* `last_48h_email_sent_at` (TIMESTAMP, NULLABLE) — idempotency guard for the email cron.

#### Table: `checklist_item_states`
* `id` (UUID, Primary Key)
* `dispute_id` (UUID, FK → `disputes.id`, ON DELETE CASCADE) — uses the surrogate UUID key, not the natural `stripe_dispute_id`.
* `item_index` (INTEGER)
* `is_completed` (BOOLEAN, default `false`)

## Business Rules

### BR-01: Timezone Processing [FSD-REQ-013]
All transaction deadlines are stored and computed in UTC on the backend. The client UI parses the current browser timezone strictly to render the countdown for user clarity. The email cron reads `stripe_connections.timezone` (default `UTC`) to fire at 08:00 local time.

### BR-02: Status Escalation Path [FSD-REQ-014]
When the API sync engine detects that a Stripe dispute `status` has transitioned away from `needs_response`, the local status is automatically set to `Submitted` (if transitioned to `under_review`) or archived (if transitioned to `won` / `lost`), and the checklist is locked.

### BR-03: Zero PII Guarantee [FSD-REQ-015]
No platform service shall extract, receive, or persist individual client parameters such as full names, email addresses, credit card numbers, billing addresses, or items purchased. All diagnostic references must use `stripe_dispute_id` or non-sensitive card metadata (`card_brand`, `card_last4`).

### BR-04: Automated Checklist Verification [FSD-REQ-016]
If a user marks all active checklist items complete, the internal status helper toggles `status_local` from `Drafting` to `Ready to Submit`. Reversible manually.

## Edge Cases & Error Handling

### EC-01: Revoked Merchant Authorization [FSD-REQ-017]
* **Scenario:** Merchant revokes access credentials in Stripe.
* **System Action:**
    1. Detect `401 Unauthorized` responses from Stripe.
    2. Flip `stripe_connections.active` to `false`.
    3. Show high-priority banner in the dashboard: *"Stripe authentication has expired or been revoked. Re-authenticate to resume data sync — or continue entering disputes manually."*
    4. Suspend automated sync jobs and email alerts for this profile until re-authenticated.

### EC-02: API Rate Limits [FSD-REQ-018]
* **Scenario:** Stripe returns HTTP 429 during sync.
* **System Action:**
    1. Retry with exponential backoff + jitter (delays: 1s, 4s, 16s).
    2. If still failing after 3 attempts, pause requests for that connection, log a warning (structured console log — external observability like Sentry deferred to post-MVP), resume on next cron cycle.

### EC-03: Null Deadline Handling [FSD-REQ-019]
* **Scenario:** Stripe returns a null `evidence_details.due_by`.
* **System Action:**
    1. Render the deadline display as `Pending Date`.
    2. Show helper message: *"Stripe has not yet set a submission deadline for this dispute."*
    3. Exclude from active countdown calculations and 48-hour email notifications.

### EC-04: Concurrent Modification Conflict [FSD-REQ-020]
* **Scenario:** Multiple users edit the same dispute's checklist simultaneously.
* **System Action:**
    1. Last-Write-Wins (LWW) strategy per checklist row.
    2. Save checklists incrementally per checkbox (not on a form submit) to minimize conflict windows.

### EC-05: Duplicate Manual Entry [FSD-REQ-023] *(NEW)*
* **Scenario:** A user manually enters a dispute with a `stripe_dispute_id` that a later Stripe sync also finds.
* **System Action:**
    1. On sync, if a `stripe_dispute_id` already exists as a manual entry, update the existing row rather than create a new one.
    2. Set `source = 'stripe'` after the sync overwrites the manual data (so we retain "how it originally arrived" only until it's been reconciled). Preserve local checklist state.

### EC-06: Stripe API Downtime [FSD-REQ-024] *(NEW)*
* **Scenario:** Stripe's API is unavailable during a scheduled sync (5xx errors).
* **System Action:**
    1. Log the failure per connection.
    2. Retry the entire sync at the next scheduled cron (do not queue immediate retries beyond the FSD-REQ-018 backoff — the daily cron is soon enough).
    3. Users can still use the dashboard, enter disputes manually, and toggle checklist state during downtime.
