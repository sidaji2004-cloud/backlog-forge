# Product Requirements Document: Chargeback Tracker MVP

## Overview

The Chargeback Tracker is a specialized B2B SaaS web application designed for operations associates at direct-to-consumer (DTC) brands generating $500K–$5M in annual revenue. Currently, these brands manage chargebacks through fragmented methods — checking multiple email accounts, checking Stripe dashboards, or maintaining manual spreadsheets. This fragmentation leads to missed dispute deadlines and lost revenue.

This MVP integrates with Stripe (via OAuth) to sync open disputes, provides a manual-entry path as a first-class fallback for merchants whose Stripe setup blocks OAuth, calculates and displays hard deadlines with visual countdowns, and provides interactive, reason-code-specific evidence checklists. The goal is a single, highly actionable dashboard that ensures no winnable dispute is lost to administrative oversight.

**Success looks like:** Olivia connects Stripe (or enters her first dispute manually) within 3 minutes of signup, sees the countdown UI updating in real time, and marks her first checklist item complete before closing the tab.

---

## Goals & Non-Goals

### Goals
*   **Centralize and Track:** Sync all open Stripe disputes into a single web dashboard within 60 seconds of account connection — OR ingest a manually-entered dispute in under 90 seconds.
*   **Prevent Missed Deadlines:** Display a highly visible, accurate countdown timer for every dispute based on Stripe's `evidence_details.due_by` epoch timestamp (or the user-entered deadline for manual disputes).
*   **Guide Evidence Collection:** Provide dynamic, interactive checklists mapped to Stripe's primary dispute reason codes, with a generic fallback for unmapped codes.
*   **Zero PII Risk:** Build a lightweight architecture that stores zero sensitive credit card numbers or raw customer personally identifiable information (PII).

### Non-Goals
*   **No Multi-Processor Support:** We will not support processors other than Stripe (e.g., Braintree, Adyen, PayPal) in the MVP.
*   **No Direct Evidence Submission via API:** Users will not upload evidence files to our platform to submit them to Stripe. The MVP redirects users to the exact Stripe dashboard URL to submit finalized evidence.
*   **No Direct Shopify API Integration:** We will not pull order or shipping data directly from Shopify in v1. Users will reference Shopify manually.

## User Personas

### Olivia, Operations Associate
*   **Profile:** Olivia manages customer service and order fulfillment at a growing DTC apparel brand. She is detail-oriented but perpetually context-switching.
*   **Pain Points:** She misses 2–3 chargeback deadlines a month because Stripe notifications get buried in her inbox. When she does dispute them, she is unsure exactly what evidence Visa/Mastercard want for "Product not received" vs. "Unrecognized transaction."
*   **Goal:** She wants a clear daily view showing exactly what chargebacks are due next and what files she needs to gather from Shopify and ShipStation to win them.

## Features & Requirements

### 1. Stripe OAuth Connection & Initial Sync
*   **Description:** Self-serve onboarding that authenticates the user's Stripe account and performs a first-time sync of active disputes.
*   **Requirements:**
    *   Use Stripe Connect (OAuth) to request read-only access to the user's disputes (`r_disputes` permission scope).
    *   Upon successful connection, pull all disputes where `status` is `needs_response`.
    *   Map Stripe dispute objects to our local database schema. Do not store sensitive transaction details or raw billing names; only store: `stripe_dispute_id`, `amount`, `currency`, `reason`, `status`, `due_by_timestamp`, and `card_brand_last4`.
    *   **Failure handling:** if OAuth fails or the merchant is on a Stripe Connect / Shopify Payments configuration that blocks the standard flow, surface a clear error message and offer the Manual Dispute Entry path (Feature 6) as an alternative.

### 2. Prioritized Chargeback Dashboard
*   **Description:** The primary screen displaying all active, unresolved disputes.
*   **Requirements:**
    *   Display a clean table of open disputes sorted chronologically by nearest deadline (`due_by` date).
    *   **Data fields to display:**
        *   Dispute ID / Reference (derived from Stripe or entered manually)
        *   Amount (displayed in the dispute's native currency, e.g., `$120.00 USD` or `€95.00 EUR`). When sorting by amount, the dashboard uses the raw dispute amount within each currency — **no cross-currency conversion in MVP**. A per-currency subtotal at the top of the dashboard is deferred to post-MVP.
        *   Reason Code (translated from Stripe's snake_case to readable text, e.g., `subscription_canceled` → "Canceled Subscription")
        *   Status Badge (e.g., "Needs Response", "Drafting", "Evidence Submitted")
        *   Deadline Countdown (visual indicator)

### 3. Countdown Timer Engine
*   **Description:** Calculates and formats the precise time remaining for evidence submission.
*   **Requirements:**
    *   Calculate time remaining as due-by time minus current time.
    *   **Formatting Rules:**
        *   If remaining time is less than 24 hours: Display in hours and minutes (e.g., `14h 32m`) in **bold red text**.
        *   If remaining time is between 24 hours and 5 days: Display in days and hours (e.g., `2d 11h`) in **orange text**.
        *   If remaining time is 5 days or more: Display in days (e.g., `8 days`) in **gray text**.
    *   If the time expires, change status to "Expired" and move to the history tab.

### 4. Reason-Code Evidence Checklists
*   **Description:** Interactive checklists mapped to the specific reason code provided by Stripe (or selected during manual entry).
*   **Requirements:**
    *   When a user clicks on a dispute row, open a detailed panel showing the customized checklist.
    *   Checklists must match the requirements of the major card networks (Visa, Mastercard, Amex).
    *   **For any reason code not explicitly mapped in the table below, show a generic 4-item checklist** (order receipt, customer communication, delivery/fulfillment proof, refund policy) **plus a dropdown allowing the user to override with any of the mapped codes** if they know which one applies. This handles new codes Stripe adds over time and unusual edge codes.
    *   **Mapping Table:**

| Stripe Reason Code | Display Name | Core Checklist Items Required |
| :--- | :--- | :--- |
| `product_not_received` | Product Not Received | Tracking number & carrier name; proof of delivery matching order address; copy of fulfillment notification email |
| `unrecognized` | Unrecognized Transaction | Proof of customer IP address match at checkout; copy of order confirmation email; proof of prior successful transactions from same customer (if any) |
| `subscription_canceled`| Canceled Subscription | Copy of subscription terms shown at checkout; proof of customer cancellation date & time; log of customer account usage post-cancellation date |
| `merchandise_not_as_described` | Not as Described | Product description from website; return policy copy; proof that customer did not return item, or that return instructions were not followed |
| *(any other code)* | Generic Evidence Checklist | Order receipt / invoice; direct customer communication (email chain); delivery / fulfillment proof; refund policy as shown at checkout |

### 5. Progress Tracking & Stripe Integration Deep-Linking
*   **Description:** Simple workflow persistence allowing Olivia to track her preparation and quickly jump to Stripe to complete the task.
*   **Requirements:**
    *   Allow users to toggle a dispute's local status to "Drafting" or "Ready to Submit".
    *   Store checked/unchecked state for each checklist item in our database, tied to the user account.
    *   Display a prominent button: **"Go to Stripe Dashboard to Submit"**.
    *   **Deep-link schema:** `https://dashboard.stripe.com/disputes/{{stripe_dispute_id}}`. This redirects the user directly to the action page in their authenticated Stripe session. See FSD-REQ-010 for Connect-account handling.

### 6. Manual Dispute Entry *(NEW)*
*   **Description:** A first-class alternative to OAuth ingestion. Any user, at any time, can add a dispute by hand.
*   **Requirements:**
    *   Prominent **"+ New Dispute"** button on the dashboard (visible whether or not Stripe is connected).
    *   Form fields: Stripe Dispute ID (string), Amount + Currency, Reason Code (dropdown of the 4 mapped codes plus "Other — generic checklist"), Due-By date + time in UTC.
    *   On save: create a `disputes` row with `status_stripe = 'needs_response'`, `status_local = 'Open'`, seed the checklist based on the chosen reason code, redirect back to the dashboard.
    *   Manually-entered disputes are treated identically to Stripe-synced disputes for countdown, checklist, and deep-linking (deep-link uses the Stripe Dispute ID the user provided; if invalid, they land on Stripe's own error page, which is acceptable for MVP).
    *   *Why this is a Must:* the OAuth path can fail for merchants on Stripe Connect / Shopify Payments configurations. Manual entry is the escape hatch that guarantees every merchant gets value on day 1.

### 7. Win-Rate Reporting (basic) *(NEW)*
*   **Description:** Consumes the 90-day post-submission polling data (see Open Question 2) and surfaces it as a simple report.
*   **Requirements:**
    *   New **Reports** tab in the app.
    *   Bar chart: wins vs. losses grouped by reason code, over the last 90 days.
    *   Total-recovered-revenue figure at the top.
    *   No filters or drill-downs in MVP — one page, one chart.
    *   Data source: the `status_stripe` field on disputes that have transitioned from `needs_response` → `won` / `lost` during the 90-day polling window.

## Prioritization (MoSCoW)

**Must:** Stripe Connect OAuth (Read-Only); sync active disputes (`needs_response`); Manual Dispute Entry (fallback path); dynamic countdown timer engine; reason-code checklist generator (with generic fallback); deep-linking to Stripe dispute URLs.

**Should:** Save state of checklist items; "Drafting" local status state; Win-Rate Reporting (basic).

**Could:** 48-hour email alert notifications (highest-value cut if timeline slips); manual text notes field per dispute; "Ignore/Archive" dispute action; total winnable revenue dollar metrics.

**Won't:** Automated PDF dispute package compiler; direct Stripe API evidence uploading; multi-merchant login/switching profiles; cross-currency aggregation / conversion.

---

## Out of Scope

To guarantee delivery within the 4-week timeline, the following features are strictly out of scope for the MVP:
1.  **Direct Uploads:** We will not accept file uploads (JPG, PDF, PNG) to our servers. All document uploading must occur on Stripe's native platform.
2.  **Payment Processors other than Stripe:** Adyen, PayPal, and Authorize.Net are excluded from this release.
3.  **Automatic Dispute Resolutions:** The system will not automatically reply to disputes on behalf of the merchant.
4.  **Historical Dispute Ingestion:** We will not fetch or display disputes that were resolved or closed prior to the user's registration date.

## Open Questions

1.  **Stripe API Webhook Delay:**
    *   *Question:* Should we rely purely on manual sync buttons, or do we need to implement Stripe Webhooks (`chargeback.created`, `chargeback.updated`) immediately for the MVP?
    *   *Impact:* Implementing webhooks increases backend complexity but guarantees real-time notification accuracy for our 48-hour email alerts.
    *   *Proposed Answer:* For the 4-week MVP, we will run a daily background sync job at 02:00 UTC for all connected accounts, and provide a manual "Sync Now" button on the dashboard. Fully asynchronous webhooks will be deferred to v1.1.

2.  **Tracking Win/Loss Status Post-Dispute:**
    *   *Question:* How reliably can we track the outcome of a dispute to measure our users' win rates?
    *   *Impact:* Stripe updates dispute statuses to `won` or `lost` weeks after submission. If the user stops logging into our app, we might miss the resolution data.
    *   *Proposed Answer:* The background sync job will continue to fetch updates for any dispute marked as "Submitted" for up to 90 days. This data feeds the Win-Rate Reporting screen (Feature 7), which makes the polling visible and demoable — no orphaned backend work.

3.  **Stripe Dispute Link Construction:**
    *   *Question:* Does Stripe's URL pattern `dashboard.stripe.com/disputes/dp_XXXX` work reliably across all Stripe account configurations (including sub-accounts and Express/Custom accounts)?
    *   *Impact:* If the deep-link fails, users will land on a 404 page, degrading the core workflow.
    *   *Proposed Answer:* Yes, for standard accounts. For merchants using Connect with managed accounts, we prepend the `stripe_merchant_account_id` per FSD-REQ-010. We will confirm account types during onboarding.
